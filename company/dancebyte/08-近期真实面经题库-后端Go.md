## 7. 实战题库升级版（标准答案 + 追问 + 反问陷阱）

### Q1：说一下 GMP 模型，为什么轻量？

- 标准答案：G 是 goroutine，M 是内核线程，P 是执行上下文与本地队列。调度器通过 P 将大量 G 复用到少量 M 上，并配合 work-stealing 降低线程切换成本。轻量的核心是 goroutine 初始栈小、按需增长，且调度在用户态完成。
- 高频追问：如果一个 goroutine 长时间执行 CPU 密集逻辑，会有什么影响？
- 追问要点：可能导致同 P 上其他 G 饥饿；Go 1.14+ 有异步抢占缓解，但仍建议在长循环中检查 `context` 或拆分任务。
- 反问陷阱：直接说“goroutine 一定比线程快”。更准确是“在高并发 IO/短任务场景下，goroutine 的调度与内存成本通常更优”。

### Q2：channel 有缓冲和无缓冲怎么选？

- 标准答案：无缓冲 channel 强同步，发送和接收必须配对，适合做协程间协作信号；有缓冲 channel 允许短期削峰，适合生产消费解耦。选型依据是是否需要严格同步语义、峰值突刺大小、可接受延迟。
- 高频追问：缓冲区是不是越大越好？
- 追问要点：不是。过大可能掩盖下游故障并增加延迟抖动，应结合吞吐、处理时延、内存占用压测确定。
- 反问陷阱：把 channel 当消息队列替代品长期堆积数据。应保持“进程内短链路通信”定位。

### Q3：map 并发读写为什么会出问题？

- 标准答案：Go 原生 map 不是并发安全结构，并发写或读写会触发运行时检测并 panic（fatal error），因为扩容和桶迁移期间结构状态可能被竞争破坏。
- 高频追问：什么时候用 `sync.Map`？
- 追问要点：读多写少、key 集合相对稳定、并发高时可考虑；若写频繁或需要复杂复合操作，通常 `map + RWMutex` 可控性更高。
- 反问陷阱：认为 recover 能兜住 map 并发 panic。`fatal error` 通常不可恢复，会导致进程退出。

### Q4：你怎么设计幂等接口？

- 标准答案：客户端生成幂等键，服务端在“业务唯一约束 + 幂等表（或 Redis）”上做去重，保证同一幂等键多次请求只落一次库。关键是把“幂等判定”和“业务写入”放在同一事务边界。
- 高频追问：幂等键过期策略怎么定？
- 追问要点：按业务重试窗口和资金/订单风险定 TTL，高风险业务保留更久并做落库审计。
- 反问陷阱：只在网关层做去重。网关只能挡一部分重复，最终一致性仍要靠业务层兜底。

### Q5：分布式事务为什么常用 Saga，不用 2PC？

- 标准答案：微服务跨库场景下 2PC 对可用性和性能影响较大，协调器故障风险高。Saga 通过本地事务 + 补偿事务实现最终一致性，吞吐更高、耦合更低，更适合互联网业务。
- 高频追问：补偿失败怎么办？
- 追问要点：补偿也要幂等；引入重试队列、死信队列、人工介入流程，并记录全链路状态机。
- 反问陷阱：说“Saga 没有一致性问题”。Saga 是最终一致，不是强一致。

### Q6：慢 SQL 排查流程是什么？

- 标准答案：先定位现象（QPS、RT、CPU/IO），再用慢日志和 `EXPLAIN ANALYZE` 看执行计划，重点关注 type、rows、extra、回表与临时表；随后通过索引优化、SQL 改写、分批处理验证收益，最后做回归压测。
- 高频追问：加索引后为什么可能更慢？
- 追问要点：选择性差、回表成本高、优化器误判、写放大增加导致整体变慢。
- 反问陷阱：遇慢就“先加联合索引”。应先确认查询模式和过滤选择性。

### Q7：缓存击穿怎么治理？

- 标准答案：热点 key 过期瞬间大量请求打到 DB。常见治理：互斥重建（singleflight/分布式锁）、逻辑过期 + 异步刷新、热点永不过期 + 主动失效。
- 高频追问：分布式锁如何防误删？
- 追问要点：value 使用唯一 token，释放时 compare-and-del；长任务加 watchdog 续期。
- 反问陷阱：把穿透、击穿、雪崩混为一谈，面试官通常会继续追问边界。

### Q8：kube-proxy 的 iptables 和 ipvs 差别？

- 标准答案：两者都做 Service 四层转发。iptables 规则链匹配，规则多时性能下降明显；ipvs 基于内核虚拟服务表，转发效率与扩展性更好，适合大规模 Service 场景。
- 高频追问：那为什么还有团队用 iptables？
- 追问要点：部署简单、兼容性成熟、规模不大时足够；历史包袱和运维习惯也影响选型。
- 反问陷阱：说“ipvs 全面替代 iptables”。实际还受 CNI、内核、集群规模与历史兼容约束。

### Q9：Pod Pending 你会怎么查？

- 标准答案：先 `kubectl describe pod` 看 events，再看调度失败原因（资源不足、亲和冲突、污点不容忍、PVC 未绑定、镜像拉取策略）。然后到节点和存储侧交叉验证，最后确认配额与 LimitRange。
- 高频追问：资源看着够，为什么还调度失败？
- 追问要点：碎片化、requests/limits 配置过高、拓扑约束或 device plugin 资源不足都可能导致“看起来够但不可用”。
- 反问陷阱：只看 `top node` 就下结论。调度依据是 requests 和约束，不是瞬时使用率。

### Q10：最严重线上故障你怎么处理？

- 标准答案：按“止血优先、再定位、后复盘”执行。先降级/限流/回滚控制影响面，再基于监控与日志定位根因，最后输出复盘：直接原因、根本原因、监控缺口和防复发动作。
- 高频追问：怎么证明改进有效？
- 追问要点：给出可量化指标（MTTR、错误率、SLO 违约时长）和演练结果（故障注入通过率）。
- 反问陷阱：只讲“大家加班修好了”。面试官更看重机制建设与复用能力。
