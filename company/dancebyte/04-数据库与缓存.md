# 数据库与缓存面试题（含参考答案）

## 1. MySQL 索引失效常见原因？
**答：**对索引列做函数/计算、最左前缀不满足、类型隐式转换、范围后继续匹配、like 前缀通配。

## 2. 事务隔离级别与幻读？
**答：**MySQL InnoDB 默认可重复读（RR），通过 MVCC + next-key lock 降低幻读风险。

## 3. 为什么慢查询会拖垮系统？
**答：**占用连接与 I/O，放大锁等待，导致线程堆积和连锁超时。

## 4. Redis 适合哪些场景？
**答：**热点缓存、分布式锁、计数器、会话、限流、排行榜、短期状态存储。

## 5. 缓存穿透/击穿/雪崩怎么防？
**答：**
- 穿透：布隆过滤器 + 空值缓存  
- 击穿：互斥锁/单飞机制  
- 雪崩：过期时间随机化 + 多级缓存 + 限流降级

## 6. 双写一致性怎么做？
**答：**优先“先写 DB 再删缓存”，结合延迟双删和重试队列保障最终一致。

## 7. 分库分表后主键怎么设计？
**答：**用全局唯一 ID（雪花算法/号段），避免自增主键跨库冲突。

## 8. 如何设计高并发读写模型？
**答：**读写分离、热点键拆分、批量写入、异步化、幂等写、削峰填谷。

## 9. PostgreSQL 和 MySQL 怎么选？
**答：**复杂查询、JSONB、扩展生态偏 PG；通用 OLTP 与成熟运维场景 MySQL 更常见。

## 10. 数据库性能优化优先级？
**答：**先 SQL 与索引，再架构（分库分表/读写分离），最后才是硬件扩容。

## 高频追问（含回答要点）

### 追问 1：缓存和数据库数据不一致怎么解释？
**要点：**
- 分布式场景只能追求最终一致
- 给出一致性窗口与补偿策略
- 关键写路径必须幂等可重试

### 追问 2：什么时候不该上 Redis 缓存？
**要点：**
- 低复用、低命中数据
- 强一致且不可容忍脏读
- 访问模式不稳定、维护成本高于收益

### 追问 3：如何判断是索引问题还是锁问题？
**要点：**
- 先看执行计划和慢 SQL
- 再看事务等待与锁监控
- 最后看连接池与线程堆积
