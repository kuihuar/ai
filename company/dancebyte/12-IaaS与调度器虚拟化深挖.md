# IaaS 与调度器/虚拟化深挖题（第12版）

> 说明：聚焦岗位描述中的“控制面/数据面微服务、资源调度引擎、云计算 IaaS 经验”。
> 结构：每题包含 标准答案 + 高频追问 + 反问陷阱。

---

## Q1：你如何理解 IaaS 控制面与数据面的边界？

- 标准答案：控制面负责资源模型、生命周期编排、策略与权限；数据面负责真实流量和实例运行时路径。控制面强调一致性与可审计，数据面强调性能与稳定性，两者通过 API 和状态同步解耦。
- 高频追问：为什么要严格做控制面/数据面解耦？
- 追问要点：避免故障相互放大；控制面变更不应直接冲击业务流量；可独立扩容和故障域隔离。
- 反问陷阱：把“控制面也要承载业务流量”当常态。多数平台设计里这是反模式。

## Q2：资源调度器的核心目标是什么？

- 标准答案：在约束条件下做最优或次优分配，平衡吞吐、时延、成本、公平性与可用性。关键流程是 Filter（可行性）+ Score（优先级）+ Bind（落位）。
- 高频追问：为什么调度不可能绝对最优？
- 追问要点：问题本质接近 NP-hard，线上需要在有限时间做近似最优，并持续重调度。
- 反问陷阱：只谈算法，不谈工程约束（调度时延、可解释性、回滚）。

## Q3：Kubernetes 调度扩展常见做法有哪些？

- 标准答案：优先使用原生调度能力（亲和、污点、优先级、抢占），再考虑 Scheduler Framework 插件扩展；批处理场景可引入 Volcano/Kueue。先“少改内核、多用扩展点”。
- 高频追问：什么时候值得自研调度插件？
- 追问要点：当业务约束无法用原生能力表达，且收益能覆盖长期维护成本时。
- 反问陷阱：一开始就 fork scheduler。升级和兼容成本通常被低估。

## Q4：如何做多租户资源公平与高优先级保障？

- 标准答案：配额 + 优先级 + 抢占协同。租户维度做基础保障，闲时可借用；关键业务设置高优先级并可抢占低优先级任务，配合 PDB 避免系统抖动。
- 高频追问：抢占会造成什么副作用？
- 追问要点：低优任务反复被打断，吞吐下降；需配合最小运行时间和重试退避策略。
- 反问陷阱：只要公平不要优先级。生产环境必须允许“关键业务优先”。

## Q5：虚拟机在 K8s 上（KubeVirt）创建路径怎么讲？

- 标准答案：用户提交 VM 对象后，virt-controller 协调生成 VMI，调度到节点，virt-handler 在节点准备运行环境，virt-launcher 拉起 QEMU/KVM 实例，随后网络（Multus/CNI）与存储（CSI/DataVolume）挂载完成，探针就绪后对外服务。
- 高频追问：VMI 和 VM 的区别？
- 追问要点：VM 是期望状态抽象，VMI 是一次运行实例；重启/迁移会带来新 VMI 生命周期。
- 反问陷阱：把 virt-launcher 当“控制器”。它是节点侧运行承载，不负责全局编排决策。

## Q6：Multus、CNI、NMState 在虚拟机网络里各做什么？

- 标准答案：CNI 负责主网络连接；Multus 提供多网卡编排能力；NMState 负责节点网络配置声明式管理（如 bridge/bond/VLAN 前置准备）。三者是“节点网络准备 + Pod/VM 接入 + 多网扩展”的分层关系。
- 高频追问：什么时候必须用 NMState？
- 追问要点：需要节点级网络拓扑变更时，如创建 Linux bridge、bond、VLAN trunk，且要可回滚可审计。
- 反问陷阱：认为 Multus 能替代节点网络配置。Multus 只负责编排附加网络，不做底层网卡改造。

## Q7：Longhorn/CSI 在 VM 存储链路中的职责？

- 标准答案：CSI 负责标准化卷供应、挂载、扩容接口；Longhorn 作为 CSI 实现提供分布式块存储与副本高可用。VM 场景常把系统盘、数据盘都映射为 PVC，再由 KubeVirt 挂载为磁盘设备。
- 高频追问：为什么卷扩容后 VM 内空间没变？
- 追问要点：卷块设备扩容后还需客体 OS 内执行分区/文件系统扩展。
- 反问陷阱：把“存储类支持扩容”理解为“业务自动可用扩容完成”。

## Q8：虚拟机在线迁移（Live Migration）关键约束是什么？

- 标准答案：要求共享或可迁移存储、网络可达一致、CPU 特性兼容，并控制脏页速率。迁移过程通常是预拷贝 + 短暂停机切换，目标是最小业务抖动。
- 高频追问：什么情况下迁移会失败？
- 追问要点：脏页过高、目标节点资源不足、网络策略阻断、存储不可达或 CPU model 不兼容。
- 反问陷阱：把迁移当“纯控制面动作”。实际是控制面 + 节点运行时 + 网络存储协同。

## Q9：如何设计 IaaS 平台的配额体系？

- 标准答案：按层级定义配额：租户、项目、环境（dev/stage/prod），维度覆盖 CPU、内存、存储、IP、负载均衡、快照数。配额策略要支持硬限额和软告警，并提供超额审批流程。
- 高频追问：配额为什么还要“预留池”？
- 追问要点：应对突发和故障迁移，避免关键业务因资源耗尽无法恢复。
- 反问陷阱：只设硬限额不做告警与治理，最终会变成频繁人工救火。

## Q10：控制器 Reconcile 设计有哪些关键原则？

- 标准答案：幂等、可重入、最小变更、状态机清晰、失败可重试。每次循环都以“观察现状 -> 对齐期望 -> 更新状态”为核心，并通过 finalizer 保障删除语义完整。
- 高频追问：如何避免 Reconcile 热循环打爆 apiserver？
- 追问要点：指数退避、事件去重、只在状态变化时更新、控制并发 worker 数。
- 反问陷阱：在 Reconcile 中直接做重 I/O 串行阻塞，导致队列堆积与级联超时。

## Q11：如何做云平台控制面的高可用？

- 标准答案：无状态服务多副本部署，关键状态落在高可用存储（如 etcd/DB），通过 leader election 控制单活控制器；入口层做负载均衡和健康检查，变更链路加审计与回滚。
- 高频追问：为什么控制器常常“多副本 + 单 leader”？
- 追问要点：多副本保可用，单 leader 保证写路径一致性，避免并发控制冲突。
- 反问陷阱：把所有组件都做 active-active 写入。没有冲突解决机制会引入双写风险。

## Q12：一次典型的“资源调度异常”你会怎么复盘？

- 标准答案：先还原时间线（请求、排队、调度、绑定、启动），再按层定位（配额/调度策略/节点资源/运行时依赖），区分“策略问题”与“容量问题”。最后沉淀为：监控指标、阈值告警、自动化修复和演练脚本。
- 高频追问：调度指标你最关注哪些？
- 追问要点：调度成功率、调度时延分位数、队列积压深度、抢占次数、Pending 时长分布。
- 反问陷阱：复盘只给结论不给证据。没有指标和事件链就难以服众和复用。
