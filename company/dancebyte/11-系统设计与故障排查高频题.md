# 系统设计与故障排查高频题（标准答案 + 追问 + 反问陷阱）

> 说明：这版聚焦后端 Go 面试中的“架构设计 + 稳定性 + 故障处理”。
> 目标：帮助你在项目深挖轮回答出“可落地、可量化、可复盘”。

---

## Q1：如何设计一个支撑百万 QPS 的 API 网关？

- 标准答案：采用无状态多副本 + L4/L7 负载均衡 + 本地缓存 + 限流熔断。数据面尽量无共享，控制面统一下发路由和策略；灰度发布与动态配置必须可回滚。容量上按峰值 * 冗余系数规划，并预留突发缓冲。
- 高频追问：限流放网关还是业务服务？
- 追问要点：两层都要有。网关做全局保护和租户级配额，业务层做关键接口保护，避免绕过网关时失控。
- 反问陷阱：只强调“加机器就行”。没有流量治理和熔断，扩容会被级联故障吞掉收益。

## Q2：如何做高并发场景的缓存架构设计？

- 标准答案：采用“本地缓存 + Redis”两级缓存；热点 key 预热，过期时间随机抖动，互斥重建防击穿；写路径采用延迟双删或 binlog 异步修正。关键是定义一致性等级与可接受陈旧时间。
- 高频追问：为什么不是强一致缓存？
- 追问要点：强一致代价高且吞吐受限，多数业务可接受短暂最终一致；核心资金/库存链路再使用强约束方案。
- 反问陷阱：把所有数据都缓存。低命中或高变更数据会放大复杂度与成本。

## Q3：如何设计订单系统的幂等与防重？

- 标准答案：请求层使用幂等键，存储层使用唯一约束，状态机层保证重复请求返回同结果。消息消费端做去重表或业务主键去重。核心在“判重与写入同事务边界”。
- 高频追问：用户重复点击和消息重复消费怎么统一？
- 追问要点：统一落到“业务唯一标识 + 状态机幂等”模型，入口和异步都依赖同一约束。
- 反问陷阱：只在前端防重。前端不能覆盖重试、超时补发、消息重复投递。

## Q4：数据库慢查询突然暴增，你怎么在 10 分钟内处理？

- 标准答案：先止血（降级、限流、只读兜底），再定位（慢日志、执行计划、锁等待、资源瓶颈），最后修复（索引/SQL 改写/拆批）并验证。流程是“业务影响最小化优先”。
- 高频追问：怎么判断是 SQL 问题还是资源问题？
- 追问要点：看等待事件与系统指标联动：CPU/IO/锁等待/连接池；如果执行计划无变化但 IO 飙升，优先查资源和热数据分布。
- 反问陷阱：上来就重启数据库。重启会掩盖根因并增加二次风险。

## Q5：你如何设计服务降级策略？

- 标准答案：按功能分级（核心、重要、可丢弃），核心路径保留，非核心功能在高负载时自动关闭。降级触发依据错误率、P99、线程池水位，恢复依据回稳窗口。策略必须可观测、可演练、可手动接管。
- 高频追问：如何避免降级后恢复抖动？
- 追问要点：引入迟滞机制和冷却时间，避免阈值边缘来回切换。
- 反问陷阱：只设计“降”不设计“升”。没有自动恢复和阈值治理会长期处于亚健康。

## Q6：跨服务调用链故障如何定位根因？

- 标准答案：先按 traceID 聚合链路，再对齐日志与指标（请求量、错误率、延迟），定位首个异常节点；进一步看该节点依赖（DB/缓存/下游）和变更记录。最后输出根因树而不是单点结论。
- 高频追问：链路追踪采样会丢关键请求怎么办？
- 追问要点：对错误请求和慢请求提高采样率或全量采集，正常流量保留低采样控制成本。
- 反问陷阱：只看单服务日志。分布式故障常是调用链协同问题。

## Q7：如何做零停机发布与秒级回滚？

- 标准答案：发布采用灰度/金丝雀，readiness 门控流量，老版本优雅摘流；监控阈值触发自动回滚。关键是“发布系统与监控系统打通”，把回滚条件标准化。
- 高频追问：回滚会不会导致数据兼容问题？
- 追问要点：先做向后兼容发布（expand-contract），避免先发不可逆 schema；关键字段变更分阶段上线。
- 反问陷阱：把“镜像回滚成功”当“业务回滚成功”。数据与消息状态可能已前滚。

## Q8：如何设计多租户资源隔离？

- 标准答案：控制面做租户身份与权限隔离（RBAC/命名空间），资源面做配额隔离（Quota、LimitRange、节点池隔离），网络面做 NetworkPolicy。高价值租户可独立节点池或独立集群。
- 高频追问：如何兼顾隔离和资源利用率？
- 追问要点：基础配额 + 弹性借用池；高峰抢占策略要有优先级并可审计。
- 反问陷阱：只做逻辑隔离不做资源配额。容易产生“噪声租户”挤占。

## Q9：消息队列积压怎么处理？

- 标准答案：先扩消费能力和分区并行度，临时降级非核心生产流量，必要时按业务优先级分流处理。之后复盘根因：消费瓶颈、下游慢、重试风暴或消息模型不合理。
- 高频追问：盲目扩消费者为什么可能无效？
- 追问要点：可能受分区数、下游依赖、DB 写入瓶颈限制，扩容会放大竞争。
- 反问陷阱：只清积压不修模型。没有死信和重试上限，问题会反复出现。

## Q10：怎么做故障复盘才真正有价值？

- 标准答案：复盘要覆盖时间线、影响面、直接原因、根本原因、组织和流程因素、改进项及 owner/截止时间。改进项必须可验证（演练、指标、告警命中率）。
- 高频追问：如何避免复盘沦为“写文档”？
- 追问要点：所有改进项进入迭代并在下次演练验收，未完成项持续追踪。
- 反问陷阱：把“个人失误”当根因。多数 P1 故障背后是机制和流程缺陷。
