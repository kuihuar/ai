# 近期面经高频题趋势总结（Go 后端 / 云原生方向）

> 来源说明：基于近期公开面经与社区文章共性题型做归纳（腾讯云社区、Go 中文社区、SegmentFault 等公开内容）。
> 目标：帮助你把“刷题”升级为“按面试趋势准备”。

---

## 1. 近期高频考点总览（按出现频次）

1. **Go 并发与运行时**
2. **MySQL + Redis 基础与性能**
3. **微服务与分布式一致性**
4. **Kubernetes 架构与故障排查**

6. **可观测性（Prometheus/日志/Tracing）**

---

## 2. 高频问题清单（带答题关键词）

## 2.1 Go 并发与性能

- GMP 调度模型、goroutine 泄漏、channel 底层
- context 取消传播、select 使用陷阱
- map 并发安全、defer/panic/recover
- 逃逸分析、pprof 优化案例

**答题关键词**：幂等、可取消、可观测、压测对比、优化前后指标。

## 2.2 MySQL / Redis

- InnoDB 索引结构、回表、最左前缀
- 事务隔离级别、MVCC、幻读
- Redis 数据结构（zset/skiplist、hash、list）
- 缓存穿透/击穿/雪崩、分布式锁、渐进式 rehash

**答题关键词**：读写路径、瓶颈定位、一致性窗口、降级兜底。

## 2.3 微服务与分布式

- gRPC + Protobuf 为什么优于 JSON（内部服务）
- 服务治理：超时、重试、熔断、限流
- 分布式事务：Saga/TCC 取舍
- 幂等设计与消息补偿

**答题关键词**：最终一致性、业务容错、成本与复杂度权衡。

## 2.4 Kubernetes / 云原生

- 核心组件职责（apiserver/scheduler/controller/kubelet）
- CRD + Controller + Reconcile 的设计价值
- Deployment 滚动发布、Pod Pending 排查
- Helm、Operator、PaaS 化思路

**答题关键词**：声明式、状态机、Requeue、可回滚。

## 2.5 系统设计与项目深挖

- “画架构图 + 讲关键表结构 + 讲关键链路”
- 峰值流量怎么扛、性能如何优化
- 线上问题复盘：如何止血、如何防复发
- 为什么这样设计（trade-off）

**答题关键词**：约束条件、瓶颈识别、演进路径、风险控制。

## 2.6 可观测性（近两年明显加强）

- 关键指标体系（QPS、P95、错误率、资源利用率）
- Prometheus 告警规则设计
- 日志字段标准化（trace_id/reconcile_id）
- 指标-日志-链路联动排障

**答题关键词**：告警收敛、定位闭环、SLO 思维。

## 3. 面试官“最爱追问”的 10 个点

1. 你说优化了，怎么证明？
2. 如果重来一次，你会怎么设计？
3. 最大事故是什么？怎么止血？
4. 为什么不用 XXX 方案？
5. 一致性和可用性你如何取舍？
6. 你项目里最复杂的并发问题是什么？
7. 系统瓶颈如何定位到具体组件？
8. 如何做容量评估和压测基线？
9. 方案落地风险是什么？如何兜底？
10. 你的方案对团队协作有何影响？

## 4. 建议补充到你的面试题库中的“近期增量”

建议在现有 01~06 文件基础上新增这三类题：

1. **性能证据题**
   - “优化前后指标如何采集与对比？”
2. **稳定性事故题**
   - “出现雪崩时的 5 分钟止血动作是什么？”
3. **架构取舍题**
   - “为什么选 gRPC 不选 REST？为什么选 Kueue 不选 Volcano？”

## 5. 2 周冲刺建议（按优先级）

### 第 1 周：打基础高频
- Go 并发 + MySQL/Redis + K8s 组件
- 每天 10 题，必须口述而不是只看答案

### 第 2 周：强化场景表达
- 系统设计题 + 事故复盘题 + trade-off 题
- 每题按“背景-问题-方案-结果-反思”结构练

---

## 6. 面试作答模板（可直接套）

1. **先给结论**（一句话）
2. **讲核心原理**（2~3 点）
3. **给项目例子**（你的真实经历）
4. **说取舍与风险**（为什么这样选）
5. **补可观测性与验证指标**（怎么证明有效）

这个模板能显著提升“表达完整度”和“工程可信度”。
