# 代码审查

## 代码审查的目的

1. **保证代码质量**：发现 bug 和潜在问题
2. **知识共享**：团队成员互相学习
3. **统一规范**：确保代码符合团队规范
4. **架构一致性**：保持架构设计的一致性

## 审查检查清单

### 功能正确性
- [ ] 代码是否实现了需求
- [ ] 边界情况是否处理
- [ ] 错误处理是否完善
- [ ] 是否有潜在的 bug

### 代码质量
- [ ] 代码是否清晰易读
- [ ] 函数是否职责单一
- [ ] 是否有重复代码
- [ ] 命名是否规范

### 性能
- [ ] 是否有性能问题
- [ ] 是否有内存泄漏风险
- [ ] 数据库查询是否优化
- [ ] 并发处理是否安全

### 安全性
- [ ] 是否有安全漏洞
- [ ] 输入验证是否充分
- [ ] 敏感信息是否泄露
- [ ] 权限检查是否到位

### 测试
- [ ] 是否有足够的测试
- [ ] 测试覆盖率是否合理
- [ ] 测试用例是否全面

### 文档
- [ ] 代码注释是否充分
- [ ] 复杂逻辑是否有说明
- [ ] API 文档是否更新

## 审查流程

### 1. 提交 PR
- 提供清晰的 PR 描述
- 说明变更内容和原因
- 关联相关 issue

### 2. 审查阶段
- 审查者仔细阅读代码
- 提出具体的改进建议
- 使用友好的沟通方式

### 3. 修改完善
- 作者根据反馈修改
- 回复审查意见
- 标记已解决的问题

### 4. 批准合并
- 至少一人批准
- 通过 CI 检查
- 解决所有讨论

## 审查原则

### 对审查者
1. **建设性反馈**：提供具体的改进建议
2. **尊重作者**：使用友好的沟通方式
3. **及时响应**：尽快完成审查
4. **学习态度**：把审查当作学习机会

### 对作者
1. **虚心接受**：认真对待每个反馈
2. **及时回应**：及时回复审查意见
3. **解释说明**：对设计决策进行说明
4. **持续改进**：从审查中学习成长

## 最佳实践

1. **小 PR**：保持 PR 规模适中，便于审查
2. **自审查**：提交前自己先审查一遍
3. **自动化**：使用工具自动检查格式和规范
4. **定期总结**：定期总结审查中的常见问题
